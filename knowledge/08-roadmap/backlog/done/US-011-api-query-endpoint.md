# US-011 — API Query Endpoint (POST /api/v1/query) — Agent Orchestration Tier

**Status:** Completed & Archived · 2026-07-23

## User Story


**As a** Developer / AI Agent consumer,  
**I want to** call `POST /api/v1/query` with a natural-language question and receive a structured, synthesised answer with citations assembled by the agent orchestration tier,  
**So that** both the UI (US-010) and future machine consumers (US-037) can get complete answers — not raw retrieval chunks — through a single versioned API contract.

---

## Description

This story implements the top-level query endpoint on the agent orchestration tier (`agent/` service). It accepts a query, calls the Knowledge API (US-008) to retrieve evidence, passes the evidence through Guardrails (US-024, stub in PI-1), and synthesises a cited answer using the LLM. For PI-1 this is a single-pass synthesis (the iterative loop in US-029 is PI-2); the evaluate/re-plan nodes exist in the LangGraph graph but `should_continue` returns `False` after the first pass, with a clear TODO marking it for US-029.

---

## Business Value

- Provides the machine-readable API contract that the frontend (US-010) and future agent consumers depend on.
- Establishes the typed response contract (`answer`, `citations[]`, `trace_id`, `guardrail_flags[]`) that all downstream stories build on.

---

## Acceptance Criteria

**Given** the agent service is running and the backend Knowledge API (US-008) is accessible,  
**When** `POST /api/v1/query` is called with `{"query": "...", "requester_identity": "..."}`,  
**Then:**
- The agent calls `POST /api/v1/knowledge/query` on the backend to retrieve evidence.
- The evidence is passed to the LLM synthesis step (Gemini Flash for planning, Gemini Pro for synthesis per the model routing policy).
- The response is a typed JSON object: `{"answer": "...", "citations": [...], "trace_id": "...", "guardrail_flags": []}`.
- Each citation in the response maps to a specific `EvidenceItem` (chunk_id, source_url, content excerpt).
- `should_continue` in the LangGraph graph is hardcoded to `False` for PI-1 (single-pass); a TODO comment marks it for US-029.
- If the backend Knowledge API returns no evidence, the answer includes a clear statement that the corpus contains no relevant results.
- If the LLM call fails, the endpoint returns HTTP 503 with a structured error.
- The endpoint is protected by the internal API key (service-to-service auth).

## Non-Functional Requirements

- NFR-002 (Security) — the agent service must not hold source-system credentials; it calls the Knowledge API only.
- NFR-006 (Performance) — end-to-end latency (retrieval + synthesis) logged via `trace_id`; target p50 ≤6 seconds for single-pass MVP synthesis.
- NFR-009 (Cost optimisation) — use Gemini Flash for decomposition/planning; reserve Gemini Pro for final synthesis only.

## Dependencies

- US-008 complete (Knowledge API hybrid retrieval endpoint).
- US-024 (Guardrails) — stub/passthrough in PI-1; real integration in PI-1 if US-024 is implemented in parallel.
- LLM API key configured in secrets.

---

## Assumptions

- The agent orchestration tier is the `agent/` FastAPI service, using LangGraph for the plan → execute → respond flow.
- The existing LangGraph skeleton (`agent/app/graph.py`) is the starting point; this story upgrades the `execute` node to call US-008 and the `respond` node to assemble citations.
- Model routing: `PLANNING_MODEL=gemini-2.5-flash`, `SYNTHESIS_MODEL=gemini-2.5-pro` — both configurable via env vars.
- The `requester_identity` is forwarded from the interface tier or provided by the machine consumer; it is not generated by the agent.

---

## Edge Cases

- **Knowledge API returns partial evidence (some sources unavailable):** Pass available evidence to synthesis; include a `source_availability_warning` in the response.
- **LLM synthesis returns a malformed or safety-blocked response:** Return HTTP 503; do not silently return an empty answer.
- **`requester_identity` missing from request:** Return HTTP 401; the endpoint must not accept anonymous queries.
- **Query is extremely long (> context window):** Truncate the evidence list to fit the context window; log the truncation event.

---

## Technical Notes / Implementation Considerations

- **LangGraph graph nodes:** `plan` (decompose query, PI-1 = no-op), `execute` (call Knowledge API), `evaluate` (PI-1 = stub returning False), `respond` (synthesise + assemble citations).
- **Citation assembly:** Map each `EvidenceItem` returned by the Knowledge API to a citation object: `{chunk_id, source_url, source_type, content_excerpt}`. Include the top-k evidence items used in synthesis, not all retrieved chunks.
- **`trace_id`:** Generate a UUID at request time; pass it through to the Knowledge API call (as a header) and include in the final response.
- **Guardrails stub:** A no-op `GuardrailsClient` that logs `"guardrails: passthrough"` and returns the input unchanged. Replaced by the real client in US-024/US-025.
- **Unit tests:** Mock the Knowledge API and LLM API; assert correct citation assembly, error handling, and response schema.
